1. Sumário executivo

A Igreja Conect é uma plataforma web interna (Single Page + SSR) que centraliza a gestão eclesiástica de uma única igreja local. Este PRD cobre o MVP, que entrega dois pilares: autenticação (login com sessão segura) e gestão de membros + discipulado + acolhimento de visitantes. O objetivo é substituir controles informais (planilhas, cadernos, WhatsApp) por um sistema único, auditável, com privacidade financeira desde o dia 1.

TL;DR em 5 bullets:

  • Membros CRUD completo de pessoas (membros ativos, congregados, visitantes), com filtros e busca textual.
  • Discipulado Vínculo de cada membro a 1 discipulador, com trava rígida de máximo 12 discípulos por líder.
  • Acolhimento Ao cadastrar um visitante, o sistema gera automaticamente um alerta para o responsável configurado pelo Admin.
  • Privacidade Histórico de dízimos visível apenas para Admin, Pastor e Financeiro — bloqueado em 3 camadas (UI, loader, service).
  • LGPD Sem CPF, sem dados fiscais; senhas com hash bcrypt; cookies httpOnly; prontos para auditoria.

A aceitação final é o próprio Admin rodando este cenário em ambiente local: "consigo cadastrar 50 membros reais, vincular todos a discipuladores respeitando o limite de 12, e gerar alertas de visitante — sem nunca ver dados financeiros de quem não deveria."

2. Personas e casos de uso

A Igreja Conect reconhece 6 perfis administrativos. Todos os 6 perfis já existem no sistema desde o MVP (o enum Cargo no banco já está preparado), mas apenas as colunas relativas a Membros têm UI funcional nesta entrega. As demais (Financeiro, Estoque) entram em sprints futuras sem quebrar schema.

2PASTORPastor titular CRUD total + visualiza dízimosFinanceiro total, Estoque somente leitura
3SECRETARIOSecretário(a) administrativo(a) CRUD totalFinanceiro (com trava de saldo), Estoque (autoriza retirada)
4DISCIPULADORLíder de célula / discipulador CRUD de membros (apenas seus discípulos)Sem acesso a financeiro

6FINANCEIROTesoureiro(a) CRUD total + visualiza dízimosFinanceiro total (com trava de saldo)

Observação crítica: os perfis 1, 2 e 6 (Admin, Pastor, Financeiro) são os únicos que podem visualizar a aba "Fidelidade Financeira" no perfil do membro. No MVP a aba existe na UI, mas os dados de dízimos não são populados (módulo Financeiro está fora de escopo). Quando o Financeiro entrar em sprint futura, a trava de visibilidade já estará aplicada desde o dia 1.

2.1 Casos de uso principais do MVP

  • UC-01 — Login: Pastor digita email e senha, sistema valida hash bcrypt, abre sessão, redireciona para o painel autenticado.
  • UC-02 — Listar e filtrar membros: Secretário consulta a lista de membros, aplica filtros (tipo, ministério, discipulador) e busca por nome.
  • UC-03 — Cadastrar visitante: Qualquer perfil autenticado cadastra uma pessoa como VISITANTE. Sistema gera alerta automático para o responsável configurado pelo Admin.
  • UC-04 — Vincular discípulo: Discipulador associa um membro a si mesmo. Sistema bloqueia o 13º discípulo com mensagem clara.
  • UC-05 — Configurar acolhimento: Admin define um Membro ou Ministério como responsável padrão pelo acolhimento de novos visitantes.
  • UC-06 — Marcar alerta como lido: Responsável pelo acolhimento acessa a Central de Alertas e marca o alerta do novo visitante como lido/resolvido.

3. Escopo do MVP

O MVP entrega 3 blocos: autenticação, módulo de membros e central de alertas. Tudo o que não está listado abaixo é explicitamente fora de escopo (ver §4).

3.1 Autenticação

  • Login com email + senha. Senha armazenada como hash bcrypt em Membro.senhaHash.
  • Sessão via cookie httpOnly assinado, sem JWT e sem localStorage. Decisão confirmada pelo usuário no briefing.
  • Logout invalida a sessão no servidor (registro removido) e limpa o cookie.
  • Middleware de proteção nas rotas autenticadas: anônimos são redirecionados para /login.
  • Seed do primeiro Admin via Prisma, idempotente (rodar 2x não duplica).
  • Senhas nunca trafegam em logs, nunca retornam em payloads de API, nunca aparecem na hidratação SSR.

3.2 Módulo de Membros

O módulo entrega a ficha cadastral unificada, segmentação por tipo, vínculo de discipulado, gestão de ministérios, aba de fidelidade financeira (bloqueada por perfil) e configuração de acolhimento de visitantes.

3.2.1 Sub-funcionalidades

  • CRUD completo de membros (criar, listar, detalhar, editar, excluir). Qualquer perfil autenticado pode escrever.
  • Ficha cadastral com dados pessoais + eclesiásticos + endereço. Sem CPF, profissao e estadoCivil opcionais.
  • Segmentação por tipo: MEMBRO_ATIVO, CONGREGADO, VISITANTE. A transição entre tipos é 100% manual — o sistema não conta presença automaticamente.
  • Listagem com filtros: por tipo, por ministério, por discipulador, e busca textual por nome.
  • Vínculo de discipulado (Gerenciador de Vínculo) com 2 travas rígidas: 1 discípulo → 1 discipulador; 1 discipulador → máximo 12 discípulos ativos.
  • Vinculação a ministérios (relação N:N). Ex: louvor, infantil, diaconal.
  • Configuração de destino de visitantes (somente Admin): aponta 1 Membro ou 1 Ministério como responsável pelo acolhimento.
  • Aba "Fidelidade Financeira" presente no perfil do membro, mas bloqueada/oculta para perfis sem permissão. No MVP, renderiza estado vazio "Módulo Financeiro ainda não disponível".

3.2.2 Tipos de membro (transição manual)

TipoSignificadoQuem registraComo se move
VISITANTEPrimeira visita, ainda sem vínculoQualquer perfil autenticadoManual → CONGREGADO
CONGREGADOFrequenta mas ainda não é membro plenoLíder / SecretárioManual → MEMBRO_ATIVO

3.2.3 Gerenciador de Vínculo de Discipulado

  • 1 discípulo → exatamente 1 discipulador (relação 1:N).
  • 1 discipulador → até 12 discípulos. Tentar o 13º é bloqueado com mensagem clara e amigável.
  • Auto-vínculo bloqueado (não posso ser meu próprio discipulador).
  • Loop bloqueado (A discípulo de B, B discípulo de A) — anti-ciclo.

3.2.4 Ministérios

  • Cadastro simples (nome + descrição).
  • Vínculo N:N entre Membro e Ministério (uma pessoa pode estar em vários ministérios).
  • Admin pode configurar 1 Ministério como responsável pelo acolhimento de visitantes (alternativa ao Membro responsável).

3.2.5 Configuração de acolhimento (somente Admin)

  • Aponta 1 Membro ou 1 Ministério como responsável pelo acolhimento de novos visitantes.
  • Esta configuração é o gatilho que faz a Central de Alertas receber o aviso sempre que alguém cadastrar um visitante.

3.2.6 Aba "Fidelidade Financeira"

  • Componente restrito acoplado ao perfil do membro.
  • Listagem decrescente de dízimos (mais recente primeiro).
  • Visível apenas para ADMIN, PASTOR, FINANCEIRO.
  • No MVP, a aba existe na UI mas renderiza o estado "Módulo Financeiro ainda não disponível" — os dados virão nas sprints futuras.

3.3 Central de Alertas

A Central de Alertas é a única forma de comunicação interna do MVP. Não há push, não há e-mail. Tudo passa pela tela.

  • Tela única que lista os alertas do usuário logado.
  • No MVP, o único gatilho é "novo visitante cadastrado" (RN-MEM-05).
  • Ações: marcar como lido e marcar como resolvido.
  • Sem priorização, sem agrupamento, sem filtros no MVP (mantém escopo enxuto).

6. Privacidade e LGPD

A Igreja Conect trata dados pessoais de membros e dízimos. Por isso, segue a Lei 13.709/2018 (LGPD) com medidas técnicas verificáveis — não apenas promessas.

6.1 O que não coletamos LGPD art. 6°

  • CPF, RG, CNPJ, título de eleitor, PIS, cartão SUS — não existem no sistema. A igreja não emite nota fiscal, não faz desconto em folha, não precisa desses dados.
  • Dados fiscais, salário, capacidade financeira — idem.
  • Geolocalização, fingerprinting, analytics de terceiros — nada disso no MVP.

6.2 O que coletamos (e como protegemos)

  • Identificação: nome, data de conversão, data de batismo, tipo (membro, congregado, visitante).
  • Contato: email (único), telefone.
  • Endereço residencial: logradouro, número, bairro, cidade, estado, CEP.
  • Opcionais: profissão, estado civil.

6.3 Senhas e sessão

  • Senhas armazenadas apenas como hash bcrypt (cost 10). Nunca plain text, nunca log, nunca payload de resposta.
  • Cookie de sessão: httpOnly, sameSite=lax, secure em produção.
  • TTL sugerido de 7 dias com renovação sliding (renovado a cada request autenticado), teto de 30 dias absolutos.

6.4 Dízimos (regra de ouro)

O histórico financeiro de dízimos é considerado informação estritamente confidencial. Apenas 3 perfis veem: ADMIN, PASTOR, FINANCEIRO. Para os demais, a aba simplesmente não existe na interface — e se algum desavisado tentar acessar via URL direta, recebe 403 do servidor.

Por que essa rigidez? Um vazamento de histórico de dízimos destrói a confiança da congregação e infringe a LGPD. Por isso, o bloqueio acontece em 3 camadas independentes: a UI esconde o controle, o loader da rota checa o perfil, o service revalida o perfil. Burlar uma camada não basta — o atacante teria que burlar 3.

6.5 Direitos do titular (LGPD art. 18)

No MVP, pedidos de acesso, correção ou eliminação de dados são atendidos manualmente pelo Admin. Não há fluxo automatizado de "exercício de direitos" pelo titular; está registrado como candidato natural para sprint 2+.

7. Critérios de aceitação (Definition of Done)

O MVP é considerado pronto quando todos os itens abaixo forem verdadeiros.

7.1 Funcional DoD

  • npm run dev sobe o app, executa migrations e o seed cria 1 Admin inicial (admin@igreja.local, senha documentada).
  • É possível fazer login com o Admin e ver a lista de membros (vazia no início).
  • É possível cadastrar, listar, editar e excluir membros (CRUD completo).
  • É possível vincular 12 discípulos a um discipulador; ao tentar o 13º, o sistema bloqueia com mensagem clara.
  • Ao cadastrar um visitante, o alerta aparece na Central de Alertas do responsável configurado pelo Admin.
  • A aba "Fidelidade Financeira" some da UI para perfis SECRETARIO, DISCIPULADOR, LIDER_MINISTERIO e retorna 403 se acessada via URL direta ou API.

7.2 Qualidade (gate do phase 5) Gate

  • Cobertura de testes ≥ 85% (avaliado pelo agente tester).
  • 0 vulnerabilidades critical/high (avaliado pelo agente security).
  • planning-reviewer score ≥ 70.
  • LGPD compliant (avaliado pelo agente lgpd-officer): sem coleta de CPF, senha hasheada, cookie httpOnly, dízimos restritos por perfil, com testes que provam o bloqueio.
  • Todas as funções públicas com JSDoc completo (@param, @returns, @throws, @example).
  • Nenhum teste skipped sem justificativa no PR.

7.3 Métrica principal (north star)

"Um Admin consegue cadastrar 50 membros reais, vincular todos a discipuladores respeitando o limite de 12, e gerar alertas de visitante — sem nunca ver dados financeiros de ninguém que não deveria ver."

8. Riscos e mitigações

#RiscoProb.ImpactoMitigação
R1 Vazamento de dados de dízimos por falha em esconder a aba (RN-MEM-03) Média Alto Bloqueio em 3 camadas: UI não renderiza, loader checa perfil, service retorna 403. Teste E2E cobre o caminho direto via URL/API.
R2 RBAC mal implementado permite que um Discipulador edite membros de outra célula Média Alto Middleware de autorização em todo action/loader, com testes unitários por perfil × endpoint × operação.
R3 Migration do Prisma destrutiva em dev apaga dados de teste Alta Baixo (dev) Seed idempotente + script npm run db:reset documentado. Snapshot do dev.db antes de PR destrutivo.
R4 Vínculo de discipulado em loop (A→B, B→A) causa inconsistência Baixa Médio Validação no service: discipuladorId ≠ próprio membro e ≠ descendente na cadeia. Teste cobrindo ambos os casos.
R5 Cookie de sessão não expirando deixa usuário logado indefinidamente em máquina compartilhada Média Médio TTL de 7 dias + sliding renewal + logout server-side invalida o registro.
R6 SSR expõe dados sensíveis no HTML inicial (hidratação) Baixa Alto Loaders filtram explicitamente campos sensíveis antes de retornar. Nunca retornar senhaHash em payload.

9. Roadmap de alto nível

A sequência abaixo é indicativa, sem datas firmes. Cada sprint entrega um módulo com cobertura ≥ 85% e gate de qualidade completo.

  1. Sprint 0 (atual — MVP): Autenticação + Módulo de Membros + Central de Alertas.
  2. Sprint 1: Financeiro — caixas e lançamentos (RN-FIN-01).
  3. Sprint 2: Financeiro — transferências entre caixas + trava de saldo (RN-FIN-02, 03, 04).
  4. Sprint 3: Estoque — consumo + movimentação (RN-EST-01, 02).
  5. Sprint 4: Estoque — patrimônio + manutenção externa (RN-EST-03, 05).
  6. Sprint 5: Cron de alertas de manutenção + relatórios consolidados (RN-EST-04).

Critérios para mover de sprint (herdados do MVP): cobertura ≥ 85% • zero vulnerabilidades critical/high • LGPD compliant • planning-reviewer ≥ 70.
