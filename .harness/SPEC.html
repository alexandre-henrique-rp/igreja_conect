1. Objetivo

Esta especificação é a fonte de verdade para implementação do MVP do Igreja Conect. Cobre autenticação, módulo de membros e central de alertas — com todos os contratos de payload, regras de negócio, edge cases e testes obrigatórios.

Não-escopo explícito: módulos Financeiro, Estoque, Manutenção, Cron, e qualquer feature listada em PRD §4. Suas tabelas já existem no schema para evitar migração em sprints futuras, mas nenhum service, loader ou action é entregue no MVP.

3. Arquitetura

Monólito modular React Router 7 com SSR. Estrutura canônica de pastas (RAG architecture-monolith-modular.md): 

app/
├── root.tsx               # Layout HTML raiz, ErrorBoundary
├── routes.ts              # Config declarativa de rotas
├── app.css                # Tailwind 4 entrypoint
├── routes/
│   ├── public/            # /, /login (sem auth)
│   └── app/               # /app/** (com auth, layout autenticado)
├── api/                   # Endpoints não-página (server-only)
│   └── auth/
│       └── login.ts
├── lib/                   # Services + helpers puros (com sufixo .server.ts)
│   ├── auth.server.ts     # requireUser, requireRole, verifyCredentials
│   ├── session.server.ts  # create/get/delete/renew
│   ├── rbac.server.ts     # assertCanSeeFinancials, assertCanWriteMembers, etc.
│   ├── members.server.ts  # CRUD de Membro + transição de tipo
│   ├── discipleship.server.ts # assign + trava 12
│   ├── alerts.server.ts   # criação e listagem
│   └── money.server.ts    # formatBRLFromCents (citado p/ sprints futuras)
├── db/
│   └── prisma.server.ts   # Singleton do Prisma Client
└── components/            # Componentes React puros, sem I/O

Regras de dependência (não negociáveis, ver RAG):

  1. routes/ pode importar de lib/ e db/. Nunca o contrário.
  2. lib/ pode importar de db/. Não importa de routes/ ou components/.
  3. db/ é o único lugar que importa o PrismaClient.
  4. components/ não acessa db/. Recebe dados via props ou loader.
  5. Sufixo .server.ts em todo arquivo que toca db, process.env, bcrypt, cookie ou fs.

3.2 Decisões arquiteturais (referência)

  • ADR-001 — Session cookie httpOnly em vez de JWT (logout server-side, seguro contra XSS).
  • ADR-002 — bcryptjs em vez de bcrypt nativo (compat SSR, sem binários).
  • ADR-003 — zod para validação (sugestão, [A CONFIRMAR]).
  • ADR-004 — Monólito modular em vez de microsserviços.
  • ADR-005 — Singleton do Prisma Client via globalThis.

4.2 Subset MVP — leitura operacional

4.2.1 Membro

Campos persistidos: id (uuid), nome, tipo (default VISITANTE), cargo (nullable → sem acesso admin), email (unique, nullable), senhaHash (nullable), telefone, profissao, estadoCivil, dataConversao, dataBatismo, logradouro … cep, discipuladorId (nullable, Restrict on delete), timestamps.

Uso no MVP: cargo + senhaHash populados para os 6 perfis administrativos; nulos para membros comuns. tipo é o segmento pastoral. discipuladorId materializa RN-MEM-04 (auto-rel 1:N, máx 12 — checado no service, não no schema).

4.2.2 Ministerio + MinisterioMembro

Cadastro simples + junção N:N com @@unique([membroId, ministerioId]). onDelete: Cascade na junção é aceitável (RAG convention-prisma-sqlite.md §2.7).

4.2.3 Alerta + AlertaDestinatario

Junção N:N com lido: Boolean por destinatário. Suporta múltiplos destinatários por alerta (ex: responsável por acolhimento é um Ministério → alerta para todos os membros daquele Ministério).

4.2.4 ConfiguracaoGeral

Singleton (sempre 1 linha). Armazena responsavelVisitanteTipo (MEMBRO | MINISTERIO), e o ID do responsável (uma das colunas fica null). Service acessa via findFirst (não findUnique).

4.2.5 Session A criar — migration inicial

Sem esta tabela, não há como invalidar sessões no servidor (logout real). Estrutura sugerida (RAG architecture-monolith-modular.md e ARCH §17.2): 

model Session {
  id                  String   @id @default(uuid())
  membroId            String
  membro              Membro   @relation(fields: [membroId], references: [id], onDelete: Cascade)
  expiresAt           DateTime
  absoluteExpiresAt   DateTime
  createdAt           DateTime @default(now())
  updatedAt           DateTime @updatedAt
  @@map("sessions")
}

Requer adicionar back-relation sessions Session[] em Membro. Decisão confirmada no brief; implementação pelo backend agent da Fase 5.

4.5 Decisões de modelagem aplicadas no MVP

  • output = "../generated/prisma" (não em node_modules), gitignored.
  • onDelete: Restrict em Membro.discipulador (preserva árvore de discipulado) e em todo campo financeiro/auditoria.
  • onDelete: Cascade permitido em junção pura (MinisterioMembro, AlertaDestinatario).
  • Toda migration em dev via prisma migrate dev. Nunca db push.
  • Convenção monetária: Int em centavos (ver RAG convention-monetary-values.md). No MVP não há escrita de centavos, mas o helper formatBRLFromCents já pode existir para a aba de Fidelidade Financeira renderizar placeholder.

5. Fluxo de autenticação

5.1 Login

sequenceDiagram
  autonumber
  actor U as Pastor
  participant FE as Login form
  participant RR as React Router (action)
  participant SVC as lib/auth.server.ts
  participant BCRYPT as bcryptjs
  participant SESS as lib/session.server.ts
  participant DB as Prisma + SQLite

  U->>FE: preenche email + senha
  U->>FE: submit
  FE->>RR: POST /login (form-data)
  RR->>SVC: verifyCredentials(email, senha)
  SVC->>DB: membro.findUnique({ email })
  alt não encontrado
    SVC-->>RR: null
    RR-->>FE: 401 + formError "Credenciais inválidas"
  else encontrado
    SVC->>BCRYPT: compare(senhaPlain, membro.senhaHash)
    alt hash diverge
      BCRYPT-->>SVC: false
      SVC-->>RR: null
      RR-->>FE: 401 + formError "Credenciais inválidas"
    else hash confere
      BCRYPT-->>SVC: true
      SVC->>SESS: createSession(membro.id)
      SESS->>DB: session.create({ membroId, expiresAt, absoluteExpiresAt })
      SESS-->>SVC: sessionId
      SVC-->>RR: { user: { id, nome, cargo } }
      RR-->>FE: 302 Set-Cookie sid=...; Location=/app
    end
  end

Contrato do action /login:

  • Request: form-data email: string, senha: string.
  • Validação (Zod): email válido; senha string não vazia, máx 200 chars.
  • Resposta sucesso: 302 para /app (ou ?next=), cookie sid httpOnly.
  • Resposta falha: 401 com { formError: "Credenciais inválidas" } (não diferenciar "email não existe" de "senha errada").
  • Efeito colateral: nada é logado com PII (RAG lgpd-igreja-conect.md §2.5). Apenas safeLog({ userId, action: "login_attempt", result: "ok"|"fail" }).

5.2 Logout

sequenceDiagram
  autonumber
  actor U as Usuário
  participant FE as Header (botão Sair)
  participant RR as React Router (action)
  participant SESS as lib/session.server.ts
  participant DB as Prisma

  U->>FE: clica Sair
  FE->>RR: POST /logout
  RR->>SESS: deleteSession(sid)
  SESS->>DB: session.delete({ where: { id: sid } })
  SESS-->>RR: ok
  RR-->>FE: 302 Set-Cookie sid=; Max-Age=0; Location=/login

Contrato:

  • Request: POST /logout (CSRF seguro via cookie sameSite=lax).
  • Resposta: 302 para /login com cookie sid expirado (Max-Age=0).
  • Idempotente: se sid já não existe, responde 302 do mesmo jeito.

5.3 Middleware de proteção

Localização: app/routes/app/_middleware.ts. Executa antes de todo loader/action dentro do grupo app/. 

// app/routes/app/_middleware.ts (esqueleto)
import { redirect } from "react-router";
import { getUserFromRequest } from "~/lib/session.server";
import type { Route } from "./+types/_middleware";

export async function middleware({ request }: Route.MiddlewareArgs) {
  const user = await getUserFromRequest(request);
  if (!user) {
    const url = new URL(request.url);
    const next = encodeURIComponent(url.pathname + url.search);
    throw redirect(`/login?next=${next}`);
  }
  return { user };
}

Comportamento:

  • Anônimo em rota privada → 302 para /login?next=….
  • Autenticado em rota privada → injeta { user } em context; loaders/actions filhos leem de context.user.
  • Sessão expirada → 302 para /login?next=… (mesmo comportamento de anônimo).
  • Cache de user: o backend agent pode implementar cache em Map<sessionId, User> com TTL curto (5 min) para evitar 1 SELECT por request (RAG convention-prisma-sqlite.md §2.2).

5.4 Seed admin (idempotente)

Localização: prisma/seed.ts. Roda com pnpm prisma db seed. Padrão: 

// prisma/seed.ts (esqueleto, RAG convention-prisma-sqlite §2.8)
import { PrismaClient } from "../generated/prisma/client";
import bcrypt from "bcryptjs";

const prisma = new PrismaClient();

async function main() {
  const email = "admin@igreja.local";
  const existente = await prisma.membro.findUnique({ where: { email } });
  if (existente) {
    console.log("ADMIN já existe, seed idempotente ok.");
    return;
  }
  await prisma.membro.create({
    data: {
      nome: "Administrador",
      email,
      senhaHash: await bcrypt.hash("admin123", 10),
      tipo: "MEMBRO_ATIVO",
      cargo: "ADMIN",
    },
  });
  console.log("ADMIN seed criado.");
}

main().finally(() => prisma.$disconnect());

Registro no package.json: 

"prisma": {
  "seed": "tsx prisma/seed.ts"
}

Idempotência garantida por: findUnique({ email }) antes do create. Para tornar atômico em ambientes com múltiplos workers, usar upsert: 

await prisma.membro.upsert({
  where: { email },
  create: { /* mesmo payload */ },
  update: {}, // no-op
});

6. Fluxo de Membros

6.0 User Stories formais (acceptanceCriteria)

6 user stories derivadas do PRD §2, cada uma com critérios de aceite verificáveis em teste. Numeração US-MEM-* segue o prefixo do módulo.

US-AUTH-001 — Login com email e senha

  • Como qualquer um dos 6 perfis administrativos.
  • Quero entrar com email + senha.
  • Para acessar o painel autenticado.
  • Acceptance criteria:
    1. Submeter POST /login com email válido + senha correta → 302 para /app, Set-Cookie sid httpOnly.
    2. Submeter com senha incorreta → 401 com { formError: "Credenciais inválidas" } (não diferencia "email inexistente" de "senha errada").
    3. Cookie de sessão tem httpOnly=true, sameSite=lax, secure em prod.
    4. Após 7 dias sem atividade, sessão expira (teto absoluto 30 dias).

US-AUTH-002 — Logout invalida a sessão

  • Como usuário autenticado.
  • Quero sair do sistema.
  • Para que ninguém reuse minha sessão.
  • Acceptance criteria:
    1. POST /logout remove o registro Session no DB.
    2. Cookie sid é limpo (Max-Age=0).
    3. Requisições subsequentes com o mesmo sid resultam em 302 → /login.

US-MEM-001 — Cadastrar visitante gera alerta (RN-MEM-05)

  • Como Secretário (ou qualquer perfil autenticado).
  • Quero cadastrar uma pessoa nova como VISITANTE.
  • Para iniciar o processo de acolhimento.
  • Acceptance criteria:
    1. Submeter POST /app/membros com tipo: "VISITANTE" → 302 para /app/membros/:id.
    2. Atomically, um Alerta é criado com AlertaDestinatario apontando para o responsavelMembroId (ou para todos os membros do responsavelMinisterioId).
    3. Se a criação do alerta falhar, o visitante não é persistido (rollback da transação).
    4. Acessar GET /app/alertas como o responsável mostra o novo alerta com título "Novo visitante cadastrado".

US-MEM-002 — Vincular 12 discípulos a 1 discipulador (RN-MEM-04)

  • Como Discipulador.
  • Quero associar membros a mim como discípulos.
  • Para conduzi-los espiritualmente.
  • Acceptance criteria:
    1. POST /app/membros/:id/discipulador com { discipuladorId: meuId } → 302 OK para 1º ao 12º discípulo.
    2. Tentar vincular o 13º → 422 com mensagem "Discipulador já possui 12 discípulos ativos. Reatribua antes de vincular mais."
    3. Tentar vincular discipuladorId === discId (auto-vínculo) → 400 com mensagem clara.
    4. Tentar criar loop A→B→A → 422 com "Vínculo em loop detectado".

US-MEM-003 — Listar membros com filtros

  • Como Secretário.
  • Quero ver a lista de membros com filtros e busca.
  • Para encontrar rapidamente um cadastro.
  • Acceptance criteria:
    1. GET /app/membros?tipo=VISITANTE&q=maria retorna apenas visitantes cujo nome contém "maria" (case-insensitive).
    2. Para Discipulador, o resultado é automaticamente restrito aos seus discípulos (RBAC fina).
    3. Paginação: ?page=2&pageSize=25 retorna a 2ª página de 25 itens.
    4. Payload nunca inclui senhaHash (assertion: grep no response).

US-MEM-004 — Admin configura acolhimento (RN-MEM-05)

  • Como Admin.
  • Quero apontar 1 Membro ou 1 Ministério como responsável pelo acolhimento.
  • Para que o sistema saiba para quem enviar o alerta de visitante.
  • Acceptance criteria:
    1. GET /app/config/acolhimento é acessível a qualquer usuário autenticado (lê); a tela mostra a config atual.
    2. PUT /app/config/acolhimento só é permitido para ADMIN — outros perfis recebem 403.
    3. Payload: { responsavelVisitanteTipo: "MEMBRO" | "MINISTERIO", responsavelId: string }.
    4. Após salvar, o próximo cadastro de visitante gera alerta para o novo responsável.

US-MEM-005 — Aba "Fidelidade Financeira" bloqueada para perfis não autorizados (RN-MEM-03)

  • Como Discipulador (ou Secretário, Líder de Ministério).
  • Eu não devo ver a aba de dízimos de nenhum membro.
  • Acceptance criteria:
    1. UI: a aba Fidelidade Financeira não aparece no perfil do membro.
    2. Bypass via URL GET /app/membros/:id?tab=financeiro retorna 403.
    3. Service getDizimosByMembro chamado direto lança ForbiddenError para perfil inadequado.
    4. Teste E2E comprova os 3 caminhos (UI ausente, URL 403, service lança).

6.1 CRUD: listar com filtros e busca

Loader: app/routes/app/membros/index.tsx

// app/lib/members.server.ts (trecho)
import { prisma } from "~/db/prisma.server";
import type { Cargo, TipoMembro } from "../../generated/prisma/client";

export interface ListMembrosFilter {
  tipo?: TipoMembro;
  ministerioId?: string;
  discipuladorId?: string;
  q?: string; // busca textual por nome (case-insensitive contains)
  page?: number;
  pageSize?: number;
}

export async function listMembros(filter: ListMembrosFilter, actor: { id: string; cargo: Cargo | null }) {
  const where: any = {};
  if (filter.tipo) where.tipo = filter.tipo;
  if (filter.ministerioId) where.ministerios = { some: { ministerioId: filter.ministerioId } };
  if (filter.discipuladorId) where.discipuladorId = filter.discipuladorId;
  if (filter.q) where.nome = { contains: filter.q };

  // RBAC fina: DISCIPULADOR e LIDER_MINISTERIO só veem "seus" membros
  if (actor.cargo === "DISCIPULADOR") {
    where.discipuladorId = actor.id; // override (filtro adicional)
  } else if (actor.cargo === "LIDER_MINISTERIO") {
    where.ministerios = { some: { ministerio: { /* líder é do min */ } } };
    // Implementação completa exige model LiderMinisterio (sprint 2+).
    // No MVP, LIDER_MINISTERIO com filtro sem ministério vê lista global (escopo relaxado).
  }

  const page = filter.page ?? 1;
  const pageSize = filter.pageSize ?? 25;
  const [items, total] = await Promise.all([
    prisma.membro.findMany({
      where,
      select: MEMBRO_SAFE_SELECT, // ver §8.1 — nunca senhaHash
      orderBy: { nome: "asc" },
      skip: (page - 1) * pageSize,
      take: pageSize,
    }),
    prisma.membro.count({ where }),
  ]);
  return { items, total, page, pageSize };
}

Exemplo de URL: GET /app/membros?tipo=VISITANTE&ministerioId=abc&discipuladorId=xyz&q=maria&page=1

6.3 CRUD: atualizar e excluir

  • PUT/PATCH /app/membros/:id: mesmo schema (parcial). Service valida e atualiza. senhaHash só atualizável por rota separada (Admin ou auto-troca).
  • DELETE /app/membros/:id: bloqueado se houver discípulos vinculados (onDelete: Restrict). Erro 409 com mensagem clara. Requer desvincular antes.

6.4 Transição de tipo de membro (RN-MEM-06)

Manual. Endpoint dedicado: PATCH /app/membros/:id/tipo com payload { tipo: "CONGREGADO" | "MEMBRO_ATIVO" }.

Garantia (RN-MEM-06): o sistema não tem job nenhum que promova visitante automaticamente. Teste estático no CI: grep -rE "tipo.*=.*MEMBRO_ATIVO" app/ só pode aparecer em ações humanas (não em setTimeout, setInterval, cron, node-cron, bull, etc.).

6.9 Aba "Fidelidade Financeira" (RN-MEM-03) — bloqueada

Componente app/components/MembroFidelidade.tsx. Renderização condicional baseada em user.cargo.

7.2 Helpers obrigatórios em app/lib/rbac.server.ts

// app/lib/rbac.server.ts (esqueleto canônico)
import type { Cargo } from "../../generated/prisma/client";

export type SessionUser = { id: string; cargo: Cargo | null };

const FINANCIAL: Cargo[] = ["ADMIN", "PASTOR", "FINANCEIRO"];
const ALL_ADMIN: Cargo[] = ["ADMIN", "PASTOR", "SECRETARIO", "DISCIPULADOR", "FINANCEIRO", "LIDER_MINISTERIO"];

export function assertCanSeeFinancials(user: SessionUser): void {
  if (!user.cargo || !FINANCIAL.includes(user.cargo)) {
    throw new Response("Acesso restrito a perfis financeiros.", { status: 403 });
  }
}

export function assertCanWriteMembers(user: SessionUser): void {
  if (!user.cargo || !ALL_ADMIN.includes(user.cargo)) {
    throw new Response("Acesso restrito a usuários autenticados.", { status: 401 });
  }
}

export function assertIsAdmin(user: SessionUser): void {
  if (user.cargo !== "ADMIN") {
    throw new Response("Acesso restrito ao Admin.", { status: 403 });
  }
}

export function assertCanManageConfiguracaoGeral(user: SessionUser): void {
  if (user.cargo !== "ADMIN") {
    throw new Response("Apenas Admin pode editar configurações gerais.", { status: 403 });
  }
}

Regras de uso (RAG §4):

  • Helpers lançam Response (não retornam boolean). Loaders/actions capturam para renderizar ErrorBoundary.
  • Se 3+ if (user.cargo === "X") aparecerem espalhados, abstrair para canX(user). Regra de 3 (YAGNI).
  • Nunca confiar só na UI. assertCan* é a fonte de verdade.

11.5 e2eChains (cadeias críticas para o tester agent)

  1. Chain 1 — Login + criar visitante + alerta aparece
    • Sequence: POST /login → POST /app/membros (visitante) → GET /app/alertas do responsável
    • DataFlow: POST /login retorna Set-Cookie: sid. POST /app/membros usa esse cookie + payload com tipo: "VISITANTE", retorna 302. Transação interna cria Alerta + AlertaDestinatario (vinculado ao responsavelMembroId). GET /app/alertas filtra por membroId do usuário logado e inclui o novo alerta.
  2. Chain 2 — RBAC dízimos: 3 perfis tentam acessar aba Fidelidade
    • Sequence: POST /login (perfil X) → GET /app/membros/:id?tab=financeiro → assert 403 (se perfil não permitido) ou 200 com placeholder (se permitido)
    • DataFlow: middleware injeta user no context. Loader /app/membros/:id chama assertCanSeeFinancials(user). Se 403, o componente MembroFidelidade não recebe dados. Cross-module: user.cargo é o pivot de toda a matriz.
  3. Chain 3 — Trava de 12 discípulos: 12 passam, 13º bloqueia
    • Sequence: POST /login (DISCIPULADOR) → 12× POST /app/membros/:id/discipulador → 13ª chamada deve retornar 422 com mensagem clara
    • DataFlow: cada chamada incrementa o discipuladorId em um Membro. Service assignDisciple faz count() antes do update(). Boundary test no número 12 (passa) e 13 (falha).

11.6 crossModuleHints (dados que fluem entre módulos)

  • De POST /app/membros para GET /app/membros/:id: o campo tipo recém-criado aparece na ficha (cache: loader do detalhe refaz query, sem invalidação de cache explícita).
  • De POST /app/membros (visitante) para GET /app/alertas: o Alerta.id + Alerta.titulo + Alerta.mensagem aparecem na Central do responsável. Endpoint: GET /app/alertas lê AlertaDestinatario.membroId === user.id.
  • De PUT /app/config/acolhimento para POST /app/membros: o responsavelMembroId ou responsavelMinisterioId é lido no momento do cadastro do visitante, dentro da mesma transação.
