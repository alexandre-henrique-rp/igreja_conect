1. Objetivo

Esta especificação é a fonte de verdade para implementação do Igreja Conect. Cobre o MVP (ciclo 1) e o Módulo Financeiro (ciclo 2) — com todos os contratos de payload, regras de negócio, edge cases e testes obrigatórios.

Não-escopo explícito do MVP: módulos Estoque, Manutenção, Cron, e qualquer feature listada em PRD §4. Suas tabelas já existem no schema para evitar migração em sprints futuras, mas nenhum service, loader ou action é entregue no MVP.

Escopo do ciclo 2 (Apêndice D): implementação do Módulo Financeiro, com 5 RNs (RN-FIN-01 a 05), 8 features, 12 testes de borda e decisão pendente de Caixa.ativo.

3.1 Regras de dependência (não negociáveis)

  1. routes/ pode importar de lib/ e db/. Nunca o contrário.
  2. lib/ pode importar de db/. Não importa de routes/ ou components/.
  3. db/ é o único lugar que importa o PrismaClient.
  4. components/ não acessa db/. Recebe dados via props ou loader.
  5. Sufixo .server.ts em todo arquivo que toca db, process.env, bcrypt, cookie ou fs.

3.2 Decisões arquiteturais (referência)

  • ADR-001 — Session cookie httpOnly em vez de JWT (logout server-side, seguro contra XSS).
  • ADR-002 — bcryptjs em vez de bcrypt nativo (compat SSR, sem binários).
  • ADR-003 — zod para validação (ecossistema maduro, z.infer<>).
  • ADR-004 — Monólito modular em vez de microsserviços.
  • ADR-005 — Singleton do Prisma Client via globalThis.
  • ADR-006 (ciclo 2) — Modelagem de transferência híbrida (1 TransferenciaCaixa + 2 Lancamento espelho em $transaction).
  • ADR-007 (ciclo 2) — Trava de saldo exclusivamente no service (Camada 3 — única segurança real).

5. Fluxo de autenticação

5.1 Login — sequência

sequenceDiagram
  autonumber
  actor U as Pastor
  participant FE as Login form
  participant RR as React Router (action)
  participant SVC as lib/auth.server.ts
  participant BCRYPT as bcryptjs
  participant SESS as lib/session.server.ts
  participant DB as Prisma + SQLite
  U->>FE: preenche email + senha
  U->>FE: submit
  FE->>RR: POST /login (form-data)
  RR->>SVC: verifyCredentials(email, senha)
  SVC->>DB: membro.findUnique({ email })
  alt não encontrado
    SVC-->>RR: null
    RR-->>FE: 401 + formError "Credenciais inválidas"
  else encontrado
    SVC->>BCRYPT: compare(senhaPlain, membro.senhaHash)
    alt hash diverge
      BCRYPT-->>SVC: false
      SVC-->>RR: null
      RR-->>FE: 401 + formError "Credenciais inválidas"
    else hash confere
      BCRYPT-->>SVC: true
      SVC->>SESS: createSession(membro.id)
      SESS->>DB: session.create({ membroId, expiresAt, absoluteExpiresAt })
      SESS-->>SVC: sessionId
      SVC-->>RR: { user: { id, nome, cargo } }
      RR-->>FE: 302 Set-Cookie sid=...; Location=/app
    end
  end

Contrato do action /login:

  • Request: form-data email: string, senha: string.
  • Validação (Zod): email válido; senha string não vazia, máx 200 chars.
  • Resposta sucesso: 302 para /app (ou ?next=), cookie sid httpOnly.
  • Resposta falha: 401 com { formError: "Credenciais inválidas" }.
  • Efeito colateral: safeLog({ userId, action: "login_attempt", result: "ok"|"fail" }) (sem PII).

5.2 Logout e Middleware

Logout: POST /logout → 302 para /login com cookie expirado (Max-Age=0).

Middleware em app/routes/app/_middleware.tsx: anônimo em rota privada → 302 para /login?next=…. Autenticado → injeta { user } em context.

6. Fluxo de Membros (user stories)

6 user stories derivadas do PRD §2, cada uma com critérios de aceite verificáveis. Numeração US-* segue prefixo do módulo.

US-AUTH-001 — Login com email e senha

  • Acceptance: POST /login com credenciais válidas → 302 + cookie httpOnly. Senha incorreta → 401. TTL 7d sliding, teto 30d abs.

US-AUTH-002 — Logout invalida a sessão

  • Acceptance: POST /logout remove registro Session + limpa cookie. Requisições subsequentes com mesmo sid → 302 /login.

US-MEM-001 — Cadastrar visitante gera alerta (RN-MEM-05)

  • Acceptance: POST /app/membros com tipo: "VISITANTE" → 302 para /app/membros/:id. Atomically, um Alerta é criado para o responsável. Se alerta falhar, visitante NÃO é persistido (rollback).

US-MEM-002 — Vincular 12 discípulos (RN-MEM-04)

  • Acceptance: 1º ao 12º discípulo → OK. 13º → 422 com mensagem clara. Auto-vínculo → 400. Loop A→B→A → 422.

US-MEM-003 — Listar membros com filtros

  • Acceptance: GET /app/membros?tipo=VISITANTE&q=maria → apenas visitantes com nome contendo "maria". Para DISCIPULADOR, escopo é automaticamente os discípulos. Paginação ?page=2&pageSize=25. Payload NUNCA inclui senhaHash.

US-MEM-004 — Admin configura acolhimento (RN-MEM-05)

  • Acceptance: GET /app/config/acolhimento (lê, todos). PUT só Admin. Payload { responsavelVisitanteTipo, responsavelId }. Próximo visitante gera alerta para o novo responsável.

US-MEM-005 — Aba "Fidelidade Financeira" bloqueada (RN-MEM-03)

  • Acceptance: UI: aba NÃO aparece para SECRETARIO/DISCIPULADOR/LIDER_MINISTERIO. URL ?tab=fidelidade direta com perfil inadequado → 403. Service getDizimosByMembro lança ForbiddenError. 3 caminhos cobertos por E2E.

8. Privacidade e filtro de payload

8.1 O que nunca é persistido

  • CPF, RG, CNPJ, título de eleitor, PIS, cartão SUS (RN-MEM-02).
  • Dados fiscais, salário, capacidade financeira.
  • Geolocalização, fingerprinting, analytics de terceiros.

8.2 O que nunca é retornado em API / loader

  • Membro.senhaHash — sempre omitido via select explícito.
  • Lancamento.valorCentavos para perfis sem permissão (camada 3 + UI).
  • Endereço completo para perfis que não precisam.

8.3 Logger seguro (LGPD art. 18, 31, 37)

safeLog com allowlist: userId, action, resource, result, timestamp, ip. Proibido em logs: senha, Membro.email, Membro.telefone, Membro.endereço, Lancamento.valorCentavos.

8.5 LGPD aplicada ao Financeiro (ciclo 2)

  • Art. 6° (necessidade): apenas nome e valorCentavos em recibo de dízimo (RN-MEM-02).
  • Art. 18 (direitos do titular): anonimização de dízimo órfão (SetNull em Lancamento.membro).
  • Art. 31 (segurança): trava de saldo no service, logs sem PII, RBAC em 3 camadas.
  • Art. 37 (auditoria de leitura): backlog (não ciclo 2). Sem registro de quem viu o quê.
  • Art. 46 (responsabilidade): safeLog estruturado, sem valorCentavos.
  • Art. 48 (comunicação de incidente): processo manual (Admin notifica).

9. Validação de payload

9.1 Padrão de uso (Zod)

  • Schemas em app/lib/schemas/*.ts (um arquivo por domínio).
  • Exporta Schema (Zod) + Input (z.infer<typeof Schema>).
  • Action consome com Schema.safeParse(); em falha, retorna { fieldErrors, formErrors }.
  • Loader não valida (só lê). Validation é da borda de input (form → action).
  • Sempre .strict() em input externo (rejeita campos extras).

11.3 e2eChains (cadeias críticas)

  1. Chain 1 — Login + visitante + alerta: POST /login → POST /app/membros (visitante) → GET /app/alertas do responsável. DataFlow: cookie sid, payload visitante, transação atômica cria Alerta.
  2. Chain 2 — RBAC dízimos (3 perfis): POST /login (perfil X) → GET /app/membros/:id?tab=fidelidade → 403 ou 200. DataFlow: middleware injeta user; loader chama assertCanSeeFinancials.
  3. Chain 3 — Trava de 12 discípulos: POST /login (DISCIPULADOR) → 12× POST discipulador → 13ª chamada retorna 422. DataFlow: count + update dentro do service.

11.4 crossModuleHints (dados entre módulos)

  • De POST /app/membros para GET /app/membros/:id: campo tipo aparece na ficha.
  • De POST /app/membros (visitante) para GET /app/alertas: Alerta.id + titulo + mensagem aparecem na Central.
  • De PUT /app/config/acolhimento para POST /app/membros: responsavelMembroId lido na transação do visitante.

D.1.3 Lancamento (model)

model Lancamento {
  id              String              @id @default(uuid())
  tipo            TipoLancamento      // ENTRADA | SAIDA
  categoria       CategoriaLancamento // 7 valores (DIZIMO..TRANSFERENCIA)
  valorCentavos   Int                 // sempre > 0 (Zod)
  descricao       String
  dataCompetencia DateTime            @default(now())
  caixaId String
  caixa   Caixa  @relation(fields: [caixaId], references: [id], onDelete: Restrict)
  membroId String?   // obrigatório se DIZIMO; opcional se OFERTA; null demais (RN-FIN-05)
  membro   Membro? @relation("MembroDizimo", fields: [membroId], references: [id], onDelete: SetNull)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  @@index([caixaId, dataCompetencia])  // performance: extrato por caixa
  @@index([membroId, categoria])       // performance: aba Fidelidade
  @@map("lancamentos")
}

D.6 Trilha de auditoria (LGPD + financeiro)

D.6.1 Princípios

  • Logs estruturados via safeLog (allowlist) — nunca valorCentavos, email, telefone, endereço.
  • Carimbo do operador em TransferenciaCaixa.executadoPorId (imutável, RN-FIN-02).
  • RN-FIN-02 imutabilidade: 1 TransferenciaCaixa por operação, sem update/delete no service.
  • Dízimo órfão: SetNull em Lancamento.membro (RN-FIN-05 + LGPD art. 18).
  • Saldo reconciliável: SUM(Lancamento WHERE caixaId = X) === Caixa.saldoCentavos. Reconciliação semanal é backlog (ciclo 3+).

D.6.2 O que não vai para log

DadoRazão
senha / senhaHashLGPD + segurança (mesmo hasheada)
Membro.emailDado pessoal, PII
Membro.telefoneDado pessoal, PII
Membro.endereçoDado pessoal, PII
Lancamento.valorCentavosRN-MEM-03 + LGPD (dado financeiro sensível)

D.6.3 O que vai para log

  • userId (id do operador).
  • action (ex: "create_lancamento", "transferir_caixa", "arquivar_caixa").
  • resource (ex: "caixa:<id>", "lancamento:<id>").
  • result ("ok" | "fail").
  • timestamp (ISO 8601).
  • ip (rate limit + auditoria de origem).

D.7 Schema do SQLite (gotchas)

D.7.1 Gotcha 1 — prisma.$transaction + commit assíncrono em E2E

Prisma 7.8 tem commit assíncrono. Se o page.goto subsequente em E2E não vir o lançamento criado, usar dbSettle(100) como workaround. Ver RAG lesson-prisma-7-commit-settle-e2e.

D.7.2 Gotcha 2 — Int para centavos (nunca Float)

valorCentavos: Int cobre até R$ 21.474.836,47 (32-bit signed). Float reintroduz bugs de arredondamento (RAG convention-monetary-values). Conversão só na borda via parseBRLToCents e formatBRLFromCents.

D.7.3 Gotcha 3 — DateTime no SQLite

SQLite armazena DateTime como string ISO 8601. Prisma faz parse transparente. Fuso horário: sempre UTC no DB, conversão para America/Sao_Paulo só na exibição (helper formatDate em app/lib/format-date.ts).

D.7.4 Gotcha 4 — onDelete em TransferenciaCaixa.executadoPor é Restrict

Operador carimbado é imutável. Se um Membro deixar a igreja, executadoPorId continua apontando para ele. Histórico preservado (RN-FIN-02 + auditoria).

D.7.5 Gotcha 5 — @@index em volume esperado

@@index([caixaId, dataCompetencia]) no Lancamento otimiza extrato. @@index([membroId, categoria]) otimiza aba Fidelidade. ~1k lançamentos/mês por caixa: índice composto é suficiente (YAGNI antecipar Postgres full-text).

D.9.1 e2eChains (para o tester agent)

  1. Chain 1 — Dízimo end-to-end (métrica macro): POST /login (FINANCEIRO) → POST /app/financeiro/lancamentos (DIZIMO) → GET /app/membros/:id?tab=fidelidade (PASTOR). DataFlow: cookie sid + payload com membroId + caixaId → $transaction cria Lancamento + atualiza Caixa.saldoCentavos → aba Fidelidade lê getDizimosByMembro.
  2. Chain 2 — Transferência atômica (5 mutações): POST /login (FINANCEIRO) → POST /app/financeiro/transferencias → GET /app/financeiro/caixas/origem + GET /app/financeiro/caixas/destino. DataFlow: payload com 2 caixas + valor → assertSaldoSuficiente → $transaction cria 1 TransferenciaCaixa + 2 Lancamento + 2 Caixa.update. Extrato de ambos reflete (origem SAIDA, destino ENTRADA, mesmo valor).

D.9.2 crossModuleHints (dados entre módulos)

  • De POST /app/financeiro/lancamentos (DIZIMO) para GET /app/membros/:id?tab=fidelidade: campo Lancamento aparece na tabela (data, valor BRL, descrição).
  • De POST /app/financeiro/transferencias para GET /app/financeiro/caixas/:id: 2 lançamentos espelho com categoria: TRANSFERENCIA aparecem no extrato de cada caixa envolvido.
  • De POST /app/financeiro/lancamentos (DESPESA) para GET /app/financeiro (dashboard): Caixa.saldoCentavos decrementa e aparece no card do caixa.
